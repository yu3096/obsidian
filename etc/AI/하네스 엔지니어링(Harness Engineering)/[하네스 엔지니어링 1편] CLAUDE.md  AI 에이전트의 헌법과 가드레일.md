## 1. CLAUDE.md란?
`CLAUDE.md`는 Anthropic의 공식 CLI 에이전트인 **Claude code**가 특정 프로젝트 폴더에서 작업할 때 가장 먼저, 그리고 항상 참조하는 **전역 지침서(System Prompt)**
개발자가 프로젝트 루트 디렉토리에 이 파일을 생성해 두면, 에이전트는 코드를 작성하거나 시스템을 조작하기 전에 이 문서를 읽고 지켜야 할 규칙, 맥락, 명령어들을 스스로 학습

## 2. 하네스 엔지니어링 관점에서의 역할
하네스(Harness, 통제 환경) 관점에서 `CLAUDE.md`는 에이전트의 폭주를 막는 **가장 기본적이고 강력한 '가드레일(Guardrail)'** 역할을 수행

* **컨텍스트 자동 유지:** 매번 채팅창에 "이 프로젝트는 어떤 프레임워크를 쓰고, 테스트는 어떻게 해"라고 지시할 필요 없이, 에이전트가 프로젝트 환경을 스스로 파악하게 만듬
* **환각(Hallucination) 사전 차단:** 에이전트가 존재하지 않는 라이브러리를 임의로 사용하거나 프로젝트 컨벤션에 어긋나는 구조를 만드는 것을 시스템 레벨에서 차단.
* **자동 검증(Evaluator) 루프의 시작점:** 에이전트에게 코드를 짠 후 스스로 어떤 테스트 명령어를 실행해야 하는지 명시하여, 단순 생성을 넘어 '자기 검증(Self-Evaluation)'을 강제

## 3. CLAUDE.md의 필수 4요소
견고한 하네스를 구축하기 위해 다음 요소들을 포함하는 것이 좋음

1. **프로젝트 개요(Project Context):** 이 애플리케이션의 목적과 비즈니스 로직의 핵심을 명확히 정의
2. **기술 스택 및 아키텍처 규칙(Tech Stack & Architecture):** 사용 중인 언어 버전, 프레임워크 패턴, 상태 관리 방법 등을 엄격하게 제한
3. **실행 및 테스트 명령어(Commands):** 빌드, 린트(Lint), E2E 테스트를 위해 에이전트가 터미널에서 직접 실행해야 할 정확한 CLI 명령어를 제공
4. **금지 사항(Forbidden Actions):** 에이전트가 절대 건드려서는 안 되는 레거시 영역이나 보안상 피해야 할 안티 패턴을 명시

## 4. 실전 CLAUDE.md 템플릿 예시
아래는 하네스 엔지니어링을 적용한 `CLAUDE.md`의 표준 예시
프로젝트 상황에 맞게 수정하여 사용할 수 있음.

> **파일명:** `CLAUDE.md` (반드시 프로젝트 최상단 루트에 위치)
```text
# 프로젝트 개요
이 프로젝트는 데이터 정합성을 검증하고 대시보드 UI를 제공하는 웹 애플리케이션입니다.

# 기술 스택 및 컨벤션
- **Language:** TypeScript, Python 3.11+
- **Frontend:** React(반드시 함수형 컴포넌트와 Hooks만 사용할 것)
- **Backend:** FastAPI
- **Styling:** TailwindCSS (인라인 스타일 절대 금지)

# 하네스 규칙(절대 준수 사항)
1. 코드를 수정하기 전, 항상 `docs/architecture.md`를 먼저 읽고 전체 설계 의도를 파악할 것
2. 새로운 기능을 추가할 경우, 반드시 해당 기능을 검증하는 단위 테스트(Unit Test)를 함께 작성할 것
3. 코드를 작성한 후에는 네가 스스로 아래 명령어를 터미널에서 실행하여 작업물을 검증할 것
 - Frontend 검증: `npm run lint` 및 `npm run test`
 - Backend 검증: `pytest`
4. 만약 위 테스트 명령어의 결과가 실패(Fail)로 나온다면, 실패 로그를 스스로 분석하여 코드를 다시 수정하고 통과(Pass)할 때까지 작업을 완료하지 말 것

# 금지 사항 및 가드레일
- `legacy_api/` 폴더 내부의 코드는 절대 수정하거나 삭제하지 마라.
- 패키지를 새로 설치할 때는 임의로 진행하지 말고, 반드시 개발자에게 권한을 물어볼 것.
```

## 5. 작성 시 주의점(Best Practices)
- **백과사전이 아닌 '지도'를 주어라:** `CLAUDE.md`가 너무 길어지면 에이전트의 컨텍스트 윈도우를 지속적으로 갉아먹습니다. 100~200줄 이내로 핵심만 간결하게 작성하세요.
- **문서 분산 처리:** 방대한 기획서나 API명세는 `docs/`폴더에 따로 둔 뒤, `CLAUDE.md`에는 "자세한 컴포넌트 구조는 docs/ui_components.md를 참고해"라고 진입점(Pointer)만 적어두는 것이 효율적
- **명령어의 무결성:** 기재된 테스트나 빌드 명령어에 오타가 있으면 에이전트는 무한 에러 루프에 빠짐. 사람이 직접 터미널에서 쳐보고 완벽히 작동하는 명령어만 입력해야 함.
