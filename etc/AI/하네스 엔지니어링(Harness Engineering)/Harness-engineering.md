# AI 에이전트의 안정성을 보장하는 하네스 엔지니어링(Harness Engineering)

>**수정일:** 2026.03.31
>**작성자:** 유승민
>**참고자료:** [Open AI Harness Engineering](https://openai.com/ko-KR/index/harness-engineering/),
>		[Anthropic Harness Design for Long-Running Apps](https://www.anthropic.com/engineering/harness-design-long-running-apps),
>		[하네스 엔지니어링 - AI 에이전트 시대, 코드보다 중요한 것: Open AI vs Anthropic 하네스 전략 비교](https://goddaehee.tistory.com/565)

최근 AI 코딩 도구의 화두는 단순히 "어떤 언어 모델(LLM)이 더 똑똑한가?"를 넘어
이제는 **"아무리 좋은 모델이라도, 그 모델을 감싸는 환경(Haness)이 나쁘면 결과물도 나쁘다."** 는 사실에 주목하기 시작

## 1. 하네스(Harness)란?
하네스는 본래 말의 힘을 제어하고 방향을 잡아주는 마구(馬具)를 뜻함.
AI 맥락에서 하네스는 **LLM의 능력을 특정 작업에 맞게 제어하고, 방향을 잡아주며, 결과를 검증하는 환경과 인프라의 총체**를 의미

* **비유하자면:** 언어 모델이 CPU, 컨텍스트 윈도우가 RAM이라면 **하네스는 운영체제(OS)**
	좋은 OS 없이는 아무리 빠른 CPU도 복잡한 소프트웨어를 돌릴 수 없다.
* **구성요소:** 레포지토리 구조, 문서 체계, Lint / CI, 피드백 루프, 에이전트 간 통신 프로토콜, 입력/출력 가드레일 등
---
## 2. 프롬프트 엔지니어링 vs 하네스 엔지니어링
프롬프트 엔지니어링이 단일 에이전트에게 지시를 내리는 마이크로(Micro) 제어라면 
하네스 엔지니어링은 여러 에이전트가 협업하고 검증하는 매크로(Macro) 시스템 설계

|            | 프롬프트 엔지니어링<br>(Prompt Engineering)              | 하네스 엔지니어링<br>(Harness Engineering)                   |
| :--------: | ----------------------------------------------- | ---------------------------------------------------- |
| **핵심 목표**  | 어떻게 하면 AI가 내가 원하는 답을 내놓을까?<br>(질문의 맥락과 지시어 최적화) | AI가 낸 답이 맞는지 어떻게 검증하고 시스템화할까?<br>(도구, 피드백 루프, 구조 설계) |
| **적용 범위**  | 단일 요청/응답 수준 (Micro)                             | 전체 레포지토리 및 시스템 파이프라인 수준 (Macro)                      |
| **주요 기술**  | Few-shot, Chain-of-Thought, 역할 부여               | 다중 에이전트 루프, LLM-as-a-Judge, Linter, CI/CD            |
| **에러 대처법** | "정확하게 답변해", "틀리지 마"라고 프롬프트에 명시                  | 독립된 평가자(Evaluator)나 테스트 코드를 통한 강제 검증                 |
|  **확장성**   | 언어 모델이 바뀌면 프롬프트를 다시 깎아야 함                       | 잘 설계된 하네스는 모델의 변화를 흡수하며 진화함                          |
|   **비유**   | 좋은 시험 문제를 고민하는 **출제자**                          | 채점 기준을 만들고 자동으로 채점하는 **평가 시스템**                      |

결과적으로, 하네스 엔지니어링을 제대로 구현하기 위해서는 각 에이저트의 페르소나와 제약을 세팅하는 **프롬프트 엔지니어링 역량이 필수적인 기초 블록** 으로 사용됨.

## 3. OpenAI의 접근: 에이전트 우선 세계의 하네스
OpenAI는 수동으로 작성한 코드가 단 한 줄도 없이, 100만 줄에 달하는 소프트웨어 제품을 에이전트(Codex)로만 구축한 경험을 공유.

* **"인간은 방향을 잡고(Steer), 에이전트는 실행한다(Execute)"**
	* 엔지니어의 역할은 코딩이 아니라 환경 설계, 의도 명세, 피드백 루프 구축으로 변함
* **레포지토리 = 에이전트의 세계**
	* 에이전트가 접근할 수 없는 지식(사람 머릿속, 슬랙 대화 등)은 존재하지 않는 것과 같음
	* 모든 설계 결정은 코드 레포지토리 안(`docs/`폴더 등)에 버전 관리 되어야 함
* **백과사전 대신 '지도(Map)'를 주기**
	* 모든 규칙을 꽉 채운 거대한 `AGENTS.md`파일은 에이전트의 컨텍스트를 고갈시킴
	* 대신, `AGENTS.md`를 100줄 이내의 '목차'로 활용하여 에이전트가 필요할 때 관련 문서를 점진적으로 탐색하도록 유도
* **엔트로피 관리 (AI 슬롭 청소)**
	* 코드가 쌓일수록 품질이 저하되는 것을 막기 위해 아키텍처 불변량을 린터(Linter)로 강제
	* 백그라운드 에이전트가 주기적으로 사용하지 않는 코드나 부채를 리팩토링(Garbage Collection)하도록 구축

## 4. Anthropic의 접근: 3-Agent 구조와 평가 루프
Anthropic은 프론트엔드 UI 디자인이나 장기 실행 앱 개발 시 에이전트가 흔히 겪는 한계점을 파악하고, [GAN(적대적 생성 신경망)](https://en.wikipedia.org/wiki/Generative_adversarial_network)에서 영감을 얻은 아키텍처를 도입

* **Naive한 접근의 실패 원인**
	* **컨텍스트 불안:** 모델은 컨텍스트 윈도우가 가득 찰 수록 작업을 대충 일찍 마무리하는 경향이 있음(컨텍스트 리셋으로 해결)
	* **자기 평가의 편향:** 에이전트는 자신이 짠 코드를 맹목적으로 칭찬하는 경향이 있어서 자가 검증이 어려움
* **생성(Generator)과 평가(Evaluator)의 분리 (3-Agent 구조)**
	* **Planner:** 사용자의 짧은 요구사항을 상세한 제품 스펙으로 확장
	* **Generator:** 스펙을 바탕으로 스프린트 단위로 코드를 작성
	* **Evaluator:** 작성된 결과물을 Playwright 같은 툴을 이용하여 사용자처럼 직접 클릭해보고 채점. 에러나 디자인적 결함이 발견되면 Generator에 피드백을 통해 수정시킴
---
## 5. 실무 적용: 하네스 설계 핵심 원칙 요약
1. **지도를 주되, 백과사전을 주지 마라:** 거대한 프롬프트 대신, 분산된 문서를 찾을 수 있는 진입점만 제공
2. **생성과 평가를 분리하라:** 코드를 짜는 AI와, 이를 테스트하고 비판하는 AI를 분리하여 긴장감을 유지
3. **에이전트에게 앱을 "보여줘라":** 텍스트(코드)만 던져주지 말고, 에이전트가 직접 브라우저를 띄워 테스트할 수 있는 런타임 환경을 제공
4. **불변량은 코드로 강제하라:** 코드 스타일이나 아키텍처 의존성은 프롬프트 지시가 아닌 린터(Linter)나 CI를 통해 기계적으로 차단
5. **모델이 발전하면 하네스를 단순화하라:** LLM 자체의 성능이 올라가면, 과거 모델을 보완하기 위해 만들었던 복잡한 하네스(잦은 컨텍스트 리셋 등)를 제거하고 재검증
6. **가드레일로 안전망 구축:** 에이전트가 환각 상태에 빠져 엉뚱한 코드를 지우거나, 악의적인 프롬프트 인젝션에 휘둘리지 않도록 입출력 단계에 시스템을 보호하는 엄격한 제약 조건 설정

## 6. 어 이거 완전 회사네?
하네스 엔지니어링의 본질은 결국 **인간의 소프트웨어 개발 생명주기(SDLC) 조직을 그대로 AI 에이전트들에게 이식하는 것**.
AI도 인간처럼 실수(환각)를 하기 때문에, 분업과 상호 견제라는 검증된 시스템이 필요함.

| 실제 IT 회사 역할        | 하네스 시스템 내 역할            | 수행하는 업무 및 목적                                          |
| ------------------ | ----------------------- | ----------------------------------------------------- |
| **PM / PO** (기획)   | **Planner Agent**       | 짧은 아이디어를 구체적인 요구사항 정의서로 확장하여 개발 범위를 통제                |
| **PL / Architect** | **하네스 룰(Linter/CI)**    | "이 아키텍처 패턴을 따르라"는 규칙을 세우고, 위반 시 기계적으로 차단(불변량 강제)      |
| **실무 개발자**         | **Generator Agent**     | PM이 넘겨준 스펙과 PL이 정한 환경 내에서 실제 코드를 작성                   |
| **QA**             | **Evaluator Agent**     | 코드를 맹신하지 않고, 자동화 툴을 통해 실제 브라우저를 띄워 테스트하여 결함 찾기        |
| **DevOps**         | **Context & RAG 파이프라인** | 사내 위키(`docs/`)를 연동해주고, 작업이 끝나면 환경을 초기화(Context Reset) |
**단, 이 '완벽한 가상 회사'를 운영하기 위해 치명적인 단점도 있음**
실제 회사에서 여러 부서가 협업하면 의사소통 비용과 시간이 늘어나고, 다중 에이전트 하네스 시스템 역시 현실적인 한계에 부딪힘.

* **막대한 API 비용(Cost):** 에이전트들이 서로 코드를 리뷰하고 피드백 루프를 수십 번 돌면서 토큰(Token) 사용량이 폭발적으로 증가함. 
	 (실제 Anthropic의 실험에서 풀스택 앱 하나를 자동 구축하는 데 약 $124~$200의 비용이 발생함)
* **오래 걸리는 시간(Latency):** 에이전트가 코드를 작성한 뒤 실제 브라우저를 띄워 UI를 테스트하고 결과를 분석하는 물리적 과정을 거치기 때문에, 단일 프롬프트를 날릴 때 처럼 몇 초만에 답이 나오지 않고 최종 산출물이 나오기까지 수 시간이 걸릴 수 있음.

즉, AI에게 "너 혼자 기획, 개발, QA까지 다 해봐"라고 요구하는 대신 각자의 페르소나를 부여한 에이전트 조직을 꾸리는 것이 하네스의 핵심이지만, 
그에 합당한 **'인건비(토큰)'와 '작업 시간'**을 지불할 각오가 되어 있어야 합니다.

> [!내 생각]
    > Cost가 아무리 비싸고 Latency가 아무리 느려도 실제 사람을 여럿 쓰는 것 보다는 싸고 빠를 거 같은데...?
